# InputSwitch 菜单栏单色输入法图标设计说明

## 1. 背景

当前 `InputSwitch` 菜单栏图标已能根据当前输入法切换，但仍存在以下视觉问题：

- `ABC` 没有可用的系统输入法图标
- 部分系统输入法图标在浅色菜单栏下过深或观感不统一
- 直接依赖系统输入法原图，无法保证所有输入法在菜单栏尺寸下都有一致可读性

用户已明确放弃“彩色/圆角徽标样式”，改为采用更贴近 macOS 菜单栏习惯的 `单色 template icon` 方案。

## 2. 目标

本轮只解决菜单栏图标显示，不改输入法切换逻辑。

本轮必须达到：

- 菜单栏图标根据当前输入法变化
- 图标为单色 template 风格
- 图标自动适配浅色/深色菜单栏
- 不再依赖系统输入法原始图标的浅深色与圆角风格
- 在 `ABC`、双拼、五笔、拼音等常见输入法下保持清晰可辨

## 3. 非目标

本轮不做：

- 保留系统输入法原始图标的视觉风格
- 彩色背景或圆角徽标容器
- 自定义位图资源系统
- 菜单栏展示应用名、语言全称或复杂状态
- 输入法列表页图标风格统一

## 4. 设计方案

采用 `方案 A：用文字生成单色 template icon`。

核心思想：

- 菜单栏图标不再直接使用系统输入法原图
- 由当前输入法映射出一个短字符标记
- 使用 `NSImage` 动态绘制该字符
- 将绘制结果标记为 template image，让 macOS 自动按当前菜单栏外观渲染

该方案的优点：

- 不受系统原始输入法图标资产质量影响
- `ABC` 无图标问题自然解决
- 深色/浅色主题自动适配
- 不需要维护多套图标资源
- 菜单栏小尺寸下识别度更高

## 5. 图标语义映射

为当前输入法生成一个菜单栏短字符。

映射规则按优先级执行：

1. `ABC`、`U.S.`、英语键盘类布局  
显示：`A`

2. 双拼类输入法  
当 `displayName` 或输入法 ID 包含 `Shuangpin` / `双拼` 时，显示：`双`

3. 五笔类输入法  
当 `displayName` 或输入法 ID 包含 `Wubi` / `五笔` 时，显示：`五`

4. 拼音类输入法  
当 `displayName` 或输入法 ID 包含 `Pinyin` / `拼音` 时，显示：`拼`

5. 其它已知中文输入法  
可按名称首个高价值字符显示，例如：
- `搜狗拼音` -> `搜`
- `微信输入法` -> `微`

6. 其它未知输入法  
显示 `displayName` 的首个有效字符

7. 若仍无法提取  
回退：`⌨︎`

设计原则：

- 优先保证菜单栏可读性，而不是完全保留输入法品牌视觉
- 字符数固定为 1 个
- 不在菜单栏图标内展示多个字或缩写串

## 6. 渲染策略

### 6.1 图像生成

使用 `NSImage(size:flipped:drawingHandler:)` 或等效方式生成固定尺寸图像。

建议尺寸：

- 18x18 或接近当前菜单栏按钮适配尺寸

绘制内容：

- 一个居中的单字符 glyph
- 使用系统字体的 semibold 或 bold 字重
- 透明背景

### 6.2 Template 模式

绘制完成后必须设置：

- `image.isTemplate = true`

这样菜单栏在浅色/深色外观下会自动用系统颜色渲染，不需要代码显式判断主题。

### 6.3 回退逻辑

若图像绘制失败或字符不可用：

- 回退到纯文本 `⌨︎`

## 7. 代码落点

本轮只改以下区域：

### 7.1 输入法显示元数据

可在 `InputSourceDescriptor` 上增加一个菜单栏显示字符推导，或在菜单模型侧推导。

推荐：

- 在 `StatusMenuModel` 或独立 helper 中完成字符映射
- 避免把纯 UI 语义污染到系统输入法服务层

### 7.2 菜单模型

`StatusMenuModel` 需要从“图片 URL 或文本回退”改为：

- 单色 template 图标字符
- 或最终文本回退

### 7.3 菜单栏控制器

`StatusBarController` 增加一个小型 helper：

- 输入：菜单栏字符
- 输出：template `NSImage`

### 7.4 容器刷新

`AppContainer` 在刷新状态菜单时，只需要把当前输入法传给菜单模型。

不需要调整规则、持久化或输入法切换逻辑。

## 8. 测试策略

### 8.1 菜单模型测试

必须覆盖：

- `ABC -> A`
- `双拼 -> 双`
- `五笔 -> 五`
- `拼音 -> 拼`
- 未知输入法回退到首个有效字符
- 完全未知时回退 `⌨︎`

### 8.2 不做的测试

本轮不做：

- 像素级图像快照测试
- 浅色/深色主题截图测试

原因：

- 成本高
- 容易脆
- 当前阶段收益低于维护成本

## 9. 风险与取舍

### 9.1 品牌识别 vs. 菜单栏可读性

取舍：

- 放弃保留系统输入法原始图标品牌视觉
- 选择单字符标识

原因：

- 当前目标是菜单栏小尺寸场景下的稳定识别
- 单色 template 更符合 macOS 菜单栏习惯

### 9.2 字符映射的局限

风险：

- 某些第三方输入法无法完美用单字符表达

接受原因：

- 当前需求只要求“根据当前输入法变化”且“风格统一”
- 单字符方案足以覆盖最常用的输入法类型

## 10. 设计结论

本轮采用“单字符 + template image”的菜单栏图标方案。它用更稳定、更贴近 macOS 原生菜单栏的方式解决 `ABC` 无图标、双拼图标深浅不一致以及系统原图风格不统一的问题，同时将改动范围严格限制在菜单栏展示链路，不影响已完成的输入法切换与设置管理逻辑。
