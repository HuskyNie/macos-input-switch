# InputSwitch 设置管理与图标适配设计说明

## 1. 背景

当前 `InputSwitch` 已具备基础规则引擎、输入法系统适配、菜单栏壳层与设置窗口壳层，但仍存在以下产品缺口：

- 无法在设置页配置默认输入法
- 无法在设置页新增、编辑、删除规则
- “不管理”规则缺少稳定可见、可撤销的管理入口
- 输入法列表只能只读展示，无法参与设置
- 菜单栏图标未根据当前输入法变化，且未处理浅色/深色主题适配

本次补充设计用于在不推翻现有结构的前提下，补齐上述缺口。

## 2. 目标

本轮必须交付：

- 在“通用设置”中可设置默认输入法
- 在“输入法列表”中可将某输入法设为默认
- 在“规则列表”中可新增、编辑、删除规则
- “不管理”规则通过设置页可见、可撤销、可编辑
- 菜单栏图标跟随当前输入法变化
- 图标在浅色/深色主题下保持一致观感

## 3. 非目标

本轮不做：

- 复杂规则条件（窗口标题、URL、项目路径等）
- 批量规则导入导出
- 图标主题实时轮询监听
- 自定义图标资源系统
- 完整规则搜索、分组、分页

## 4. 交互方案

采用已经确认的 `方案 B：弹窗式规则编辑`。

### 4.1 通用设置

“通用设置”页新增：

- 默认输入法下拉选择器
- 开机启动开关
- 开机启动状态文案

默认输入法的展示规则：

- 若当前默认输入法 ID 能在可用输入法列表中匹配到，则显示对应 `displayName`
- 若匹配不到，则回退显示原始输入法 ID
- 若未设置默认输入法，则显示“未设置”

### 4.2 规则列表

规则页由两部分组成：

1. 规则摘要列表
2. 规则编辑弹窗

规则摘要列表展示：

- 应用匹配键
- 规则类型
- 若为锁定规则，则展示锁定输入法名称

每条规则提供：

- 编辑
- 删除

顶部操作提供：

- 新增规则
- 将当前前台应用带入规则草稿

### 4.3 规则编辑弹窗

弹窗字段：

- 应用名/匹配键（第一版默认只读展示，不允许自由改写匹配键）
- 规则类型：`不管理` / `锁定输入法`
- 若为锁定规则，则选择目标输入法

弹窗动作：

- 保存
- 删除
- 取消

### 4.4 输入法列表

每条输入法展示：

- 输入法名称
- 输入法 ID
- 当前是否为默认输入法

每条输入法提供：

- 设为默认

### 4.5 菜单栏图标

图标策略采用已确认的 `方案 B：统一圆角容器 + 主题适配`。

渲染优先级：

1. 使用当前输入法的系统图标
2. 若系统无图标，则回退为当前输入法文字，例如 `ABC`
3. 若当前输入法未知，则回退 `⌨︎`

视觉处理规则：

- 系统图标与文字都放入统一圆角容器
- 容器根据浅色/深色主题切换背景与前景风格
- 不直接裸用系统深色位图贴在浅色背景上

目标：

- 解决 `ABC` 没有图标的问题
- 解决双拼等系统图标在浅色主题下过深的问题
- 解决不同输入法图标无圆角、风格不统一的问题

## 5. 数据流设计

### 5.1 默认输入法

来源：

- 通用设置页下拉选择器
- 输入法列表中的“设为默认”

流转：

1. UI 触发默认输入法更新
2. `SettingsViewModel` 发出默认输入法变更回调
3. `AppContainer` 更新 `AppSettings.defaultInputSourceID`
4. `SettingsStore.save` 落盘
5. `AppContainer` 刷新 UI，并重建协调器

### 5.2 规则 CRUD

来源：

- 规则页新增按钮
- 规则页编辑弹窗
- 规则页删除按钮

流转：

1. 规则草稿在 `SettingsViewModel` 中维护
2. 保存时将草稿转换为 `AppRule`
3. `AppContainer` 更新 `AppSettings.rules`
4. `SettingsStore.save` 落盘
5. `AppContainer` 刷新 UI，并重建协调器

### 5.3 不管理规则

“不管理”不另建新系统，直接复用：

- `AppRule.ignored`

菜单栏“将当前应用标记为不管理”与设置页规则弹窗最终都写入同一个 `rules[key] = .ignored`。

### 5.4 菜单栏图标

来源：

- `TISInputSourceService.currentInputSource()` 返回的当前输入法
- 其中包含输入法名称、ID、图标 URL

流转：

1. 当前输入法变化
2. `AppContainer` 刷新菜单栏状态
3. `StatusMenuModel` 生成图标元数据
4. `StatusBarController` 渲染为系统图标或文本回退

## 6. 实现边界

本轮只改以下区域：

- `SettingsViewModel`
- `GeneralPane`
- `RulesPane`
- `InputSourcesPane`
- `StatusMenuModel`
- `StatusBarController`
- `TISInputSourceService`
- `AppContainer`

必要时新增少量测试文件或扩展现有测试文件。

本轮不改：

- `RuleEngine` 规则优先级定义
- `SwitchCoordinator` 核心决策模型
- 系统权限策略
- 更复杂的菜单栏行为

## 7. 测试策略

### 7.1 ViewModel 测试

覆盖：

- 默认输入法 ID 到显示名称的映射
- 默认输入法变更回调
- 规则新增/编辑/删除回调
- 开机启动状态文案回显

### 7.2 菜单模型测试

覆盖：

- 菜单标题序列
- 当前输入法图标存在时使用图标
- 当前输入法图标缺失时回退文本

### 7.3 回归测试

覆盖：

- 新增 `不管理` 规则后，规则列表可见
- 删除规则后，规则不再存在
- 默认输入法变更后，设置页显示名称与状态一致

## 8. 风险与取舍

### 8.1 规则编辑弹窗

取舍：

- 第一版只允许从“当前前台应用”或“已存在规则”进入编辑
- 不开放自由输入任意匹配键

原因：

- 避免用户录入无效键
- 降低规则误配概率
- 保持第一版交互简单

### 8.2 图标主题适配

取舍：

- 不尝试修改系统源图本身
- 只通过统一容器与前景/背景策略解决观感问题

原因：

- 不依赖额外资源加工
- 风险和维护成本最低
- 与当前系统图标获取路径兼容

## 9. 设计结论

本轮设计在保持现有结构不变的前提下，补齐设置管理闭环：默认输入法、规则 CRUD、不管理规则与输入法列表操作全部落到统一的 `AppSettings` 写回链路中；菜单栏图标则通过“系统图标/文字 + 统一容器 + 主题适配”解决可读性和一致性问题。该方案可以最小成本修复现有产品缺口，并为下一步继续增强规则管理体验留出空间。
