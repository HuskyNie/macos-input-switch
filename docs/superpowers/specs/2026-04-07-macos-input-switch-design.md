# macOS Input Switch 菜单栏应用设计说明

## 1. 目标

构建一个常驻 macOS 菜单栏应用，根据前台应用自动切换到该应用最近使用或被锁定的输入法，并支持识别 `iTerm2` 等在程序坞隐藏但仍可成为前台应用的进程。

核心目标：
- 常驻后台时内存占用尽量小
- 菜单栏交互保持极简
- 设置窗口提供完整规则管理
- 默认对所有应用自动记忆输入法
- 支持按应用配置“锁定输入法”和“不管理”
- 记忆粒度精确到具体输入源，支持简体全拼、双拼、五笔等细分类型

## 2. 非目标

第一版不包含以下内容：
- 按窗口标题、URL、项目路径等更细粒度规则
- 云同步、多设备同步
- 脚本插件系统
- 非系统输入源列表之外的私有切换方式
- 菜单栏内直接编辑完整规则列表

## 3. 产品形态

应用由两个部分组成：

1. 常驻菜单栏进程  
负责监听前台应用、读取与切换输入法、应用规则、维护记忆、展示极简菜单。

2. 按需打开的设置窗口  
负责展示详细状态、规则列表、输入法列表、通用设置、日志与诊断。窗口关闭后应释放不必要的 UI 状态，避免长期占用内存。

## 4. 技术方案

采用 `SwiftUI + AppKit` 的混合架构：

- `AppKit`
  - 承担菜单栏图标与菜单
  - 监听应用激活事件
  - 桥接输入法切换相关系统 API
- `SwiftUI`
  - 构建设置窗口与规则管理界面
  - 仅在用户打开设置时创建与显示
- `Accessibility`
  - 作为前台应用识别的兜底能力
- macOS 输入源 API
  - 枚举可用输入源
  - 读取当前输入源
  - 切换到指定输入源

该方案的原因：
- 原生菜单栏应用实现最顺手
- 设置页开发效率较高
- 可通过按需加载 UI 控制空闲内存
- 仍可用系统事件驱动避免轮询

## 5. 核心模块划分

### 5.1 AppLifecycle

负责应用启动、菜单栏注册、设置窗口打开关闭、登录项管理初始化。

### 5.2 ActiveAppMonitor

负责监听前台应用切换，输出统一的应用身份对象。

主识别来源：
- `NSWorkspace.didActivateApplicationNotification`

识别字段优先级：
1. `bundleIdentifier`
2. 可执行路径
3. 进程名
4. `processIdentifier` 仅作为运行时辅助信息，不作为长期规则主键

兜底策略：
- 当系统激活事件提供的信息不足时，再使用 `Accessibility` 补充聚焦应用或窗口上下文

### 5.3 InputSourceService

负责：
- 列出系统可用输入源
- 读取当前输入源
- 按输入源唯一标识切换输入法
- 监听当前输入源变化事件

记忆和规则都基于“具体输入源 ID”而不是“中文/英文”这种粗粒度标签。

### 5.4 RuleEngine

输入：
- 当前前台应用身份
- 当前输入源
- 已配置规则
- 已记忆输入源

输出：
- 是否需要切换
- 应切换到哪个输入源
- 是否应忽略本次变化

优先级固定为：
1. 不管理
2. 锁定输入法
3. 最近记忆
4. 全局默认输入法

### 5.5 MemoryStore

负责维护每个应用最近一次“用户主动使用”的输入源记忆。

### 5.6 SettingsStore

负责持久化稳定配置：
- 全局默认输入法
- 锁定规则
- 不管理规则
- 开机启动开关
- 诊断开关等通用设置

### 5.7 Diagnostics

负责记录轻量级日志，用于排查：
- 应用识别失败
- 输入源不存在
- 权限不足
- 输入法切换失败

## 6. 应用身份模型

规则绑定主键使用“应用身份”结构：

- `bundleId: String?`
- `bundlePath: String?`
- `executableName: String`
- `displayName: String`

匹配顺序：
1. 优先使用 `bundleId`
2. 若缺失，则使用 `bundlePath`
3. 若仍缺失，则使用 `executableName`

设计原则：
- 优先稳定标识，避免应用改名后规则失效
- 支持 `iTerm2` 这类隐藏 Dock 但仍可成为前台应用的场景
- 不依赖 Dock 展示状态进行识别

## 7. 输入源模型

每个输入源至少包含：

- `sourceId`
- `displayName`
- `languageCode`
- `category`
- `isSelectable`

其中 `sourceId` 为唯一主键。

示例：
- `ABC`
- `简体拼音`
- `简体双拼`
- `简体五笔`

程序内部所有规则与记忆均存储 `sourceId`，显示层再映射成人类可读名称。

## 8. 规则与记忆模型

### 8.1 规则类型

每个应用可处于以下三种状态之一：

1. `locked`
   - 锁定为指定输入源
2. `ignored`
   - 不参与自动切换，也不更新记忆
3. `memory`
   - 无显式规则，使用最近记忆；无记忆时回退全局默认

### 8.2 决策顺序

当切换到某应用时：

1. 命中 `ignored`：不做任何输入法切换，也不更新记忆
2. 命中 `locked`：切到指定输入源
3. 无显式规则但有记忆：切到该应用最近记忆输入源
4. 无规则且无记忆：切到全局默认输入源

### 8.3 记忆更新条件

只有满足以下条件时才更新记忆：
- 当前前台应用可被稳定识别
- 当前应用不在 `ignored` 规则中
- 当前应用不在 `locked` 规则中
- 本次输入源变化为用户主动触发，而不是程序自动切换引发的反射事件

## 9. 事件流设计

### 9.1 应用切换事件

1. `ActiveAppMonitor` 收到前台应用变更
2. 构造当前应用身份
3. `RuleEngine` 计算目标输入源
4. 若目标输入源与当前输入源不同，则调用 `InputSourceService` 切换
5. 标记本次切换为“程序触发”

### 9.2 输入源变化事件

1. `InputSourceService` 收到当前输入源变化通知
2. 若该事件属于程序刚刚主动切换产生的回流，则忽略
3. 否则视为用户主动切换
4. 若当前应用允许记忆，则更新该应用最近输入源

### 9.3 去抖与防环

需要一个轻量状态机，记录：
- 最近一次程序主动切换时间
- 最近一次程序主动切换的目标输入源
- 最近一次应用切换的身份摘要

用途：
- 忽略程序切换引起的输入源回流事件
- 丢弃短时间内重复的相同应用激活事件
- 避免不必要的重复切换与错误记忆

整个系统应保持事件驱动，不引入高频轮询。

## 10. 菜单栏与设置窗口交互

### 10.1 菜单栏下拉

菜单栏交互保持极简，只保留：
- 运行状态
- 当前前台应用
- 当前输入法
- 将当前应用标记为不管理
- 清除此应用记忆
- 暂停自动切换（临时）
- 打开设置
- 退出

明确不放入菜单栏的能力：
- 一步锁定当前应用到当前输入法
- 完整规则编辑
- 输入法列表管理

### 10.2 设置窗口

采用类似系统设置的侧边栏分区布局，包含：
- 当前状态
- 规则列表
- 输入法列表
- 通用设置
- 日志与诊断

设计意图：
- 将高频查看与低频配置分离
- 保持菜单栏轻量
- 保留后续扩展空间

## 11. 持久化设计

第一版使用轻量本地文件存储，不引入数据库。

建议拆分为两个文件：

1. `settings`
   - 全局默认输入法
   - 锁定规则
   - 不管理规则
   - 开机启动等通用设置

2. `memory`
   - 应用到最近输入源的映射

要求：
- 启动时一次性加载到内存
- 变更后异步原子写入
- 文件损坏时可降级恢复为默认配置

选择该方案的原因：
- 数据量很小
- 访问路径简单
- 内存占用更低
- 易于调试与导出

## 12. 权限与系统集成

第一版需要重点处理：
- 辅助功能权限提示与状态展示
- 登录项启停

行为要求：
- 缺少关键权限时，菜单栏状态应可见
- 设置页提供明确的授权说明与跳转指引
- 授权缺失时不崩溃，可继续展示配置界面

## 13. 异常处理

### 13.1 输入源不存在

场景：
- 规则或记忆引用的输入源已被用户删除

处理：
- 先尝试回退到全局默认输入法
- 若默认输入法也不可用，则保持当前输入法不变
- 记录诊断日志

### 13.2 应用无法稳定识别

处理：
- 依次尝试 `bundleId -> bundlePath -> executableName`
- 若仍无法稳定识别，则不写入记忆
- UI 中显示为未识别应用

### 13.3 输入法切换失败

处理：
- 仅记录日志并停止本次尝试
- 不做快速重试，避免事件抖动

### 13.4 配置文件损坏

处理：
- 尝试读取备份或默认配置
- 保留原始损坏文件供诊断

## 14. 性能与内存约束

为满足“后台内存占用尽量小”，必须遵守：

- 以系统通知为主，不使用高频轮询
- 设置窗口按需加载，关闭后释放窗口相关状态
- 日志缓冲有限长，避免无限增长
- 不长期持有大规模 UI 模型
- 输入源列表按需刷新，不做无意义周期扫描

空闲状态下的常驻职责仅保留：
- 应用切换监听
- 输入源变化监听
- 规则与记忆的轻量内存缓存
- 菜单栏基础状态展示

## 15. 测试策略

### 15.1 单元测试

覆盖：
- 规则优先级判定
- 输入源匹配
- 记忆更新条件
- 防环与去抖状态机

### 15.2 集成测试

覆盖：
- 配置文件加载与保存
- 应用切换事件到最终决策的链路
- 输入源回流事件被正确忽略

### 15.3 手工验证

至少验证以下场景：
- `iTerm2` 在隐藏 Dock 时仍能被识别
- 普通图形应用切换恢复上次输入法
- 锁定规则生效
- 不管理规则生效
- 用户手动把某应用从全拼改为五笔后能被记忆
- 删除某输入法后的降级处理
- 缺少权限时的提示行为
- 开机启动配置生效

## 16. 第一版范围总结

第一版必须交付：
- 常驻菜单栏应用
- 极简菜单栏下拉
- 独立设置窗口
- 前台应用识别
- 自动记忆输入法
- 锁定规则
- 不管理规则
- 全局默认输入法
- 开机启动
- 诊断日志基础能力

第一版不扩展到：
- 多条件规则系统
- 同步能力
- 高级统计与报表

## 17. 设计结论

本方案选择以轻量常驻菜单栏进程为核心，使用事件驱动方式监听前台应用与输入源变化，通过固定优先级规则实现“锁定、不管理、记忆、默认”的切换决策。设置窗口仅在需要时加载，以兼顾原生体验、可维护性和低内存占用目标。
