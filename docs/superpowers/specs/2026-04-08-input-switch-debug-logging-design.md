# InputSwitch Debug 日志开关设计说明

## 1. 背景

当前“日志与诊断”页只能看到少量状态类日志和异常日志，看不到前台应用切换、规则命中、记忆命中、忽略应用等自动切换主链路信息。这导致用户在排查“为什么切换了/为什么没切换”时缺少直接证据。

已确认根因：

- `SwitchCoordinator` 只在目标输入法不可用、默认输入法回退等异常路径写诊断日志
- 规则命中、记忆命中、不管理命中、无需切换、实际执行切换等正常路径没有任何日志
- 设置页“日志与诊断”只是展示 `DiagnosticsLogger.entries`，本身没有过滤或丢失这些日志

## 2. 目标

- 在设置中增加可持久化的 `debug` 开关
- 关闭时，保持现有重要状态/错误日志行为，避免默认日志过于嘈杂
- 打开时，额外打印自动切换调试日志
- 调试日志同时输出到：
  - 应用内“日志与诊断”页
  - 进程标准输出，便于在 Console 或终端中查看
- 调试日志至少覆盖：
  - 应用切换开始
  - 命中记忆
  - 命中规则
  - 命中不管理
  - 使用默认输入法
  - 当前已匹配无需切换
  - 实际执行切换
  - 程序触发切换后的回流事件被忽略
  - 用户主动切换后写入记忆
  - 因锁定规则或不管理而跳过记忆更新

## 3. 非目标

- 不重做日志系统，不引入分级日志框架
- 不新增日志导出、搜索、过滤能力
- 不改变现有规则决策优先级和输入法切换行为
- 不把所有现有日志都改成 debug 级别

## 4. 方案对比

### 4.1 方案 A：持久化设置项控制 debug 日志

做法：

- 在 `AppSettings` 增加 `debugLoggingEnabled`
- 在“通用设置”页增加开关
- `AppContainer` 根据该开关决定是否输出调试日志

优点：

- 符合用户心智，“设置里打开 debug”
- 开关可跨重启保留
- 改动集中在现有设置链路，维护成本低

缺点：

- 需要修改持久化模型和设置 UI

### 4.2 方案 B：诊断页内临时开关

做法：

- 只在诊断页增加开关，不写入 `AppSettings`

优点：

- 改动更小

缺点：

- 不持久
- 与“设置里的 debug 开关”目标不一致
- 关闭窗口后排查体验差

### 4.3 方案 C：始终打印主链路日志

做法：

- 不提供开关，默认所有主链路都记日志

优点：

- 实现最简单

缺点：

- 默认情况下日志过多
- 会降低普通用户阅读诊断页的信噪比

### 4.4 结论

采用方案 A。

## 5. 设计

### 5.1 数据模型

在 `AppSettings` 中新增字段：

- `debugLoggingEnabled: Bool`

默认值：

- `false`

兼容性要求：

- 旧版 `settings.json` 没有该字段时，解码后应自动回退为 `false`

实现方式：

- 为 `AppSettings` 提供自定义 `Codable` 解码逻辑，或采用能对缺失布尔字段赋默认值的等效实现

### 5.2 设置界面

在“通用设置”页新增一个开关：

- 标题：`Debug 日志`

交互要求：

- 切换后立即写回 `AppSettings`
- 打开时记录一条状态日志，说明已启用 debug 日志
- 关闭时记录一条状态日志，说明已关闭 debug 日志

视图模型要求：

- `SettingsViewModel` 增加对应的 `@Published` 状态
- 增加 `onDebugLoggingToggle` 回调
- `reload(...)` 时同步该值

### 5.3 日志分类

保留现有 `DiagnosticsLogger` 作为内存环形日志存储，不引入复杂层级；在调用层面区分两类日志：

- 普通日志：
  - 当前已有的状态日志、错误日志、保存结果日志
  - 始终写入应用内诊断页
  - 同时输出到标准输出
- Debug 日志：
  - 只在 `debugLoggingEnabled == true` 时写入应用内诊断页
  - 同时输出到标准输出

为避免接口分散，`AppContainer` 提供统一入口：

- `log(_:)` 处理普通日志
- `debugLog(_:)` 处理调试日志

### 5.4 协调器日志接入

`SwitchCoordinator` 保持规则与切换职责，不直接感知持久化设置；通过注入闭包输出调试信息。

新增注入：

- `debugDiagnostics: (String) -> Void`

记录点如下。

#### 应用切换处理

在 `handleAppDidActivate(_:)` 中记录：

- 收到应用切换事件，包含应用名、匹配键、当前输入法 ID 或名称
- 命中不管理
- 命中锁定规则
- 命中记忆
- 未命中规则，使用默认输入法
- 当前输入法已匹配，无需切换
- 目标输入法不可用并进入默认回退
- 实际调用切换前，记录将切换到哪个输入法及原因

其中“命中锁定规则 / 命中记忆 / 命中不管理 / 使用默认输入法 / 已匹配无需切换”这几个口径必须出现在文案中，便于快速扫描。

#### 输入法变化处理

在 `handleInputSourceDidChange(to:)` 中记录：

- 因程序主动切换产生的回流事件被忽略
- 当前没有活动应用，忽略记忆更新
- 当前应用为不管理，忽略记忆更新
- 当前应用命中锁定规则，不更新记忆
- 用户主动切换输入法并写入记忆成功

如果记忆写入失败，仍沿用现有错误上抛路径，由上层记录普通错误日志。

### 5.5 AppContainer 接线

`AppContainer` 负责：

- 读取和持久化 `debugLoggingEnabled`
- 将 `debugLog(_:)` 注入 `SwitchCoordinator`
- 在设置修改后重建 coordinator，保证新开关即时生效

日志输出策略：

- `log(_:)`：始终写 `DiagnosticsLogger`，并 `print`
- `debugLog(_:)`：仅当 `currentSettings.debugLoggingEnabled == true` 时写 `DiagnosticsLogger`，并 `print`

为提升可读性，debug 输出建议统一增加前缀：

- `[DEBUG]`

普通日志不强制增加前缀，以维持现有展示风格。

## 6. 文案建议

示例文案：

- `[DEBUG] 应用切换：Xcode，matchKey=bundle:com.apple.dt.Xcode，当前输入法=com.apple.keylayout.ABC`
- `[DEBUG] 命中记忆：Xcode -> im.wubi`
- `[DEBUG] 命中规则：iTerm2 -> 锁定 com.apple.keylayout.ABC`
- `[DEBUG] 命中不管理：微信，跳过自动切换`
- `[DEBUG] 未命中规则，使用默认输入法：com.apple.keylayout.ABC`
- `[DEBUG] 当前输入法已匹配，无需切换`
- `[DEBUG] 执行切换：原因=remembered，目标输入法=im.wubi`
- `[DEBUG] 忽略程序回流输入法事件：com.apple.keylayout.ABC`
- `[DEBUG] 已写入记忆：bundle:com.apple.dt.Xcode -> im.wubi`
- `[DEBUG] 当前应用命中不管理，跳过记忆更新`

## 7. 测试

### 7.1 单元测试

新增或扩展以下测试：

- `SettingsViewModelTests`
  - `reload(...)` 能映射 `debugLoggingEnabled`
  - 切换 debug 开关会触发回调
- `SwitchCoordinatorTests`
  - 命中锁定规则时会输出 debug 日志并执行切换
  - 命中记忆时会输出 debug 日志并执行切换
  - 命中不管理时会输出 debug 日志且不切换
  - 当前输入法已匹配时会输出 debug 日志且不切换
  - 用户主动切换输入法后写入记忆时会输出 debug 日志
  - 程序回流输入法事件被忽略时会输出 debug 日志
- `DiagnosticsLoggerTests`
  - 继续覆盖环形裁剪行为即可，无需改变语义

### 7.2 手动验证

1. 打开“通用设置”，确认 `Debug 日志` 开关默认关闭
2. 打开开关后，在不同应用之间切换
3. 确认“日志与诊断”页出现主链路日志
4. 确认终端或 Console 中可看到同样的 `[DEBUG]` 日志
5. 关闭开关后再次切换应用
6. 确认新的主链路 debug 日志不再追加，但普通状态/错误日志仍保留

## 8. 风险与处理

- 风险：旧配置文件缺字段导致解码失败
  - 处理：为 `AppSettings` 提供缺省值兼容逻辑
- 风险：切换事件频繁，debug 日志快速刷满内存列表
  - 处理：维持现有 `DiagnosticsLogger` 上限，仅在显式开启时打印
- 风险：在多个入口分别判断 debug 开关，后续容易遗漏
  - 处理：统一通过 `AppContainer.debugLog(_:)` 控制

## 9. 结论

本轮采用“持久化 debug 开关 + 主链路按需调试日志”的方案，以最小改动补齐自动切换的可观测性。这样既能保留当前默认的安静体验，也能在排查切换异常时同时看到应用内诊断与系统控制台输出，并明确知道当前是命中记忆、命中规则、命中不管理，还是无需切换。
